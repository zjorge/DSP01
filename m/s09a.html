Bloque 1

Ecosistema de plataformas

Realidad

No todo se construye desde cero

La mayoría de proyectos web usan alguna plataforma o framework como base. Elegir bien ahorra meses de trabajo. Elegir mal genera deuda técnica y frustración. No hay plataforma "mejor" — hay plataformas más adecuadas para cada contexto.

5 categorías

  • Builders — Webflow, Framer, Squarespace
  • E-commerce — Shopify, WooCommerce
  • CMS headless — Sanity, Contentful
  • Frameworks — Next.js, Nuxt, Astro
  • No-code — Bubble, Retool
Builders: sitios informativos, portfolios, landing pages. Personalización limitada pero rápidos. E-commerce: tiendas online, catálogos, checkout. Restricciones en checkout y temas. CMS headless: contenido dinámico, multi-canal. Requiere desarrollo frontend. Frameworks: apps complejas, SaaS, dashboards. Requiere equipo de desarrollo. No-code: MVPs rápidos, herramientas internas. Escalabilidad y performance limitadas. Cada categoría resuelve problemas diferentes. La elección depende del contexto.

Criterios de selección

  • Tipo de contenido
  • Complejidad de interacción
  • Equipo disponible
  • Presupuesto y tiempo
  • Escalabilidad
Contenido: ¿estático o dinámico? ¿Quién lo actualiza y con qué frecuencia? Complejidad: ¿informativo (leer) o transaccional (comprar, reservar, gestionar)? Equipo: ¿hay desarrolladores? ¿Solo diseñadores? ¿El cliente lo mantiene solo? Presupuesto: builders baratos al inicio, caros al escalar. Custom caro al inicio, flexible después. Tiempo: builder en 2 semanas, custom en 2-3 meses. Escalabilidad: ¿va a crecer? ¿La plataforma lo soporta? También: SEO/performance e integraciones necesarias.

Builders

  • Webflow — código limpio, CMS integrado
  • Framer — animaciones, integración Figma
  • Squarespace/Wix — templates, drag-and-drop
  • ReadyMag — editorial, creativo
Webflow: diseño visual con output de código limpio. CMS integrado. Ideal para diseñadores que quieren control sin código. Limitación: lógica compleja requiere integraciones. Framer: enfocado en interacciones y animaciones. Integración con Figma. Bueno para landing pages. CMS básico. Squarespace/Wix: templates prediseñados, editor drag-and-drop. Ideal para clientes no técnicos. Personalización limitada, código pesado. ReadyMag: diseño editorial y creativo. Portfolios, revistas digitales. No para apps ni e-commerce.
Bloque 2

E-commerce y CMS headless

Caso práctico

Shopify

Shopify es la plataforma de e-commerce más usada para tiendas pequeñas y medianas. Entender sus restricciones es clave para diseñar dentro de lo posible.

Páginas clave

  • Home — hero, categorías, propuesta
  • Colección — grid + filtros
  • Producto — imágenes, variantes, CTA
  • Carrito → Checkout → Confirmación
Home: hero, categorías destacadas, productos populares, propuesta de valor. Colección: grid de productos con filtros (precio, talla, color) y ordenamiento. Producto: imágenes, título, precio, variantes, descripción, reviews, "Agregar al carrito". Carrito: lista de productos, cantidades editables, subtotal, botón de checkout. Checkout: datos de envío, pago, resumen. En Shopify estándar, el checkout tiene diseño fijo (solo Plus permite cambios profundos). Cuenta: pedidos, direcciones, wishlist. Búsqueda: resultados, filtros, sugerencias.

Restricción

El checkout estándar tiene diseño fijo

En Shopify estándar, el checkout solo permite cambiar colores y logo, no el layout. Solo Shopify Plus (plan enterprise) permite personalización profunda del checkout. Los temas usan Liquid (lenguaje de templates propio). Las secciones son modulares. Performance depende del tema: temas pesados con muchas apps = sitio lento. Estas restricciones deben documentarse y considerarse en el diseño.

Concepto

CMS headless

Contenido separado de presentación

Un CMS headless separa el contenido (backend) de la presentación (frontend). El contenido se gestiona en el CMS y se consume vía API desde cualquier frontend. Ventajas: contenido reutilizable multi-canal, libertad total de diseño, mejor performance, escalabilidad. Desventajas: requiere desarrollo frontend, más complejidad inicial. Ejemplos: Sanity (flexible, real-time), Contentful (enterprise), Strapi (open source), Payload (TypeScript-first).

Arquitectura conceptual

Usuario → Browser
Frontend (lo que se ve)
API / Backend (la lógica)
Base de datos + CMS
Frontend: lo que el usuario ve. HTML, CSS, JS. React, Vue, Svelte. Next.js, Nuxt, Astro. Backend/API: la lógica del negocio. Autenticación, permisos, pagos, emails. El frontend le pide datos vía API. Base de datos: donde se guardan los datos. Usuarios, productos, pedidos. El diseñador debe saber qué datos existen. CMS: donde el equipo edita contenido. Puede ser parte del backend (WordPress, Shopify) o separado (headless). CDN: sirve archivos estáticos desde servidores cercanos al usuario. El diseñador no programa, pero debe entender cómo se conectan las piezas.

Bloque 3

Contenido como sistema

Concepto

Contenido no es "lo que va en la página"

Es un sistema con estructura y reglas

El contenido tiene tipos, campos, fuentes, responsables, frecuencia de cambio y restricciones. Si no defines esto, diseñas para "Lorem ipsum" y después el contenido real rompe todo.

Mapa de contenido

  • Tipo — Producto, Artículo, Evento
  • Campos — título, descripción, imagen
  • Responsable — quién lo actualiza
  • Frecuencia — diario, semanal, mensual
  • Restricciones — máx 80 chars, 16:9
Para cada tipo de contenido, definir todos estos campos. Ejemplo Producto: título (máx 80 chars), descripción (máx 500 chars), precio, imágenes (mín 1), categoría, variantes, SKU, stock. Fuente: equipo de producto. Frecuencia: semanal. Ejemplo Artículo: título (máx 100 chars), autor, fecha, cuerpo, imagen destacada (16:9), categoría, tags. Fuente: content manager. Frecuencia: 2-3 por semana. Si sabes que el título puede tener 80 caracteres, diseñas para 80 caracteres — no para "Producto X".

Regla

Si el título puede tener 200 caracteres

Diseña para 200 caracteres

El error más común: diseñar con datos ideales. "iPhone 15 Pro" cabe perfecto en la card. Pero el contenido real puede ser: "Funda protectora de silicona premium para iPhone 15 Pro Max con soporte magnético MagSafe - Color Azul Medianoche". Si no defines restricciones de longitud, el contenido real rompe el diseño. El mapa de contenido es el contrato entre diseño y contenido.

Vistas complejas

Dashboards

Alta densidad + claridad

Un dashboard muestra datos agregados y permite tomar decisiones. Es la interfaz más compleja: combina alta densidad con necesidad de claridad. Principios: priorizar por acción, KPIs arriba, filtros visibles, tablas bien diseñadas.

Principios de dashboard

  • Lo accionable arriba
  • Filtros visibles
  • Tablas con alineación correcta
  • Permisos por rol
Accionable arriba: KPIs grandes, datos que requieren acción inmediata. Detalles en tablas abajo. Filtros: los más usados siempre visibles. Los demás en "más filtros". Mostrar filtros activos. Tablas: texto a la izquierda, números a la derecha, fechas consistentes. Columnas ordenables. Hover highlight. Acciones por fila al final. Responsive: colapsar columnas secundarias o convertir a cards en móvil. Permisos: admin ve todo, editor ve su contenido, viewer solo lee. Diseñar las variaciones. Estados: vacío, loading (skeleton de tabla), error ("No pudimos cargar los datos. Reintentar.").
Bloque 4

Analítica

Principio

Diseñar sin medir es adivinar

La analítica permite saber si las decisiones de diseño funcionan. Sin datos, estás adivinando. Con datos, estás iterando con fundamento.

Tipos de eventos

  • Pageview — llegó a una página
  • Click — interactuó con algo
  • Form submit — envió formulario
  • Error — algo falló
  • Conversión — completó tarea clave
Pageview: básico pero esencial. Qué páginas se visitan y cuánto. Click: clic en CTA, link, botón específico. Mide intención. Form submit: envío de formulario. Mide conversión de formularios. Scroll depth: hasta dónde scrolleó (25%, 50%, 75%, 100%). Mide engagement. Error: formulario con error, 404, timeout. Mide fricción. Conversión: compra, registro, reserva. La métrica más importante. Engagement: tiempo en página, interacciones, vistas de contenido.

Embudo

Dónde se pierden los usuarios

Un embudo es la secuencia de pasos hacia una conversión. Ejemplo e-commerce: Home → Categoría → Producto → Agregar al carrito → Checkout → Confirmación. En cada paso se pierde un porcentaje. Medir dónde se pierden más indica dónde mejorar el diseño. Definir 8-10 eventos clave del flujo principal. Para cada uno: nombre, cuándo se dispara, qué datos incluye, por qué importa. Error común: eventos vagos. "El usuario interactúa" no es un evento. "Clic en Agregar al carrito en página de producto" sí lo es.
