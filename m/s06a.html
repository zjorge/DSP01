Bloque 1

Design systems

Concepto

Design system

No es una librería de componentes

Un design system no es solo componentes en Figma. Es un conjunto de decisiones compartidas: Principios (por qué diseñamos así), tokens (las constantes), componentes (las piezas), patrones (cómo se combinan) y documentación (cómo se usan). Una librería de componentes sin principios ni documentación es una colección, no un sistema.

Design system

  • Principios
  • Tokens
  • Componentes
  • Patrones
  • Documentación
Principios: las reglas de diseño del producto. "Claridad sobre elegancia", "Mobile-first", "Accesible por defecto". Tokens: las constantes del sistema. Colores, tipografía, espaciado, radius, sombra. Componentes: las piezas reutilizables. Botones, inputs, cards, modales. Patrones: cómo se combinan los componentes para resolver problemas. Formulario de login, flujo de checkout, dashboard. Documentación: cómo se usan correctamente. Nombre, propósito, variantes, do/don't.

Referencia

  • Material Design — Google
  • HIG — Apple
  • Polaris — Shopify
  • Primer — GitHub
Material Design: exhaustivo, opinionado, con guías detalladas por componente. Referencia para Android y web. HIG: principios + componentes nativos, menos prescriptivo. Referencia para iOS y macOS. Polaris: enfocado en e-commerce, con ejemplos de uso y anti-patrones. Excelente documentación. Primer: developer-focused, pragmático. Buena referencia para sistemas ligeros. Carbon (IBM): enterprise, tokens detallados, accesibilidad integrada. Estos son inspiración, no modelos a copiar. Tu sistema debe resolver TUS problemas.

Decisión

¿Por qué "ligero"?

Un sistema ligero no es un sistema incompleto. Es una decisión consciente de alcance. Cubre el 80% del uso real con el 20% del esfuerzo.

Decisión

80% del uso, 20% del esfuerzo

Sistema completo: 50-100+ componentes, documentación exhaustiva, equipo dedicado, meses de desarrollo. Sistema ligero: 8-15 componentes núcleo, documentación mínima suficiente, mismo equipo de producto, semanas. Ideal para: proyectos nuevos, equipos pequeños, MVPs, punto de partida que crece con el producto. Principio: empezar con lo que necesitás hoy, no con lo que podrías necesitar algún día.

Componentes núcleo

8-12

cubren el 80% de la UI

Button, Input, Select, Checkbox/Radio, Card, Alert, Toast, Modal, Navbar, Badge, Avatar, Skeleton. No todos los proyectos necesitan los 12. Elegir los que el proyecto realmente usa. Un e-commerce necesita Card y filtros. Un dashboard necesita Table y Charts. Un blog necesita Article y Sidebar. Empezar con los que usás hoy. Agregar cuando los necesités.
Bloque 2

Anatomía y propiedades

Anatomía

Qué debe tener cada componente

Cada componente del sistema debe tener una documentación mínima que permita a cualquier persona usarlo correctamente. Vamos a ver los 7 elementos que todo componente documentado necesita.

Anatomía

  • Nombre claro
  • Propósito en 1 oración
  • Variantes diseñadas
  • Propiedades configurables
  • Estados completos
  • Do / Don't
  • Accesibilidad
Nombre: "Button", no "CTA element" ni "action-box". Claro, consistente, sin ambigüedad. Propósito: "Permite al usuario ejecutar una acción principal o secundaria." Una oración. Variantes: primary, secondary, ghost, destructive. Todas diseñadas visualmente. Propiedades: las perillas. Size (sm/md/lg), con/sin ícono, ancho fijo/fluido. Estados: default, hover, focus, active, disabled, loading. Todos diseñados. Do/Don't: "Do: un solo botón primario por sección. Don't: 3 botones primarios juntos." Accesibilidad: contraste, focus visible, label para screen readers, keyboard interaction.

Propiedades

Las perillas del componente

Las propiedades son las dimensiones configurables de un componente. Deben ser: Ortogonales: cada propiedad controla una dimensión independiente. Size no afecta hierarchy. Exhaustivas: cubrir las combinaciones que se necesitan, no todas las posibles. Claras: size: sm | md | lg, no size: 1 | 2 | 3.

Propiedades

  • Ortogonales — independientes
  • Exhaustivas — para el uso real
  • Claras — sm | md | lg
Ortogonales: cambiar el size no debería cambiar el color. Cambiar la hierarchy no debería cambiar el tamaño. Cada perilla es independiente. Exhaustivas para el uso real: no diseñar todas las combinaciones posibles. Diseñar las que se usan. Regla 80/20. Claras: nombres que se entienden sin explicación. sm/md/lg > small/medium/large > 1/2/3. Ejemplo Button: hierarchy (primary/secondary/ghost/destructive), size (sm/md/lg), state (default/hover/focus/disabled/loading), iconPosition (none/left/right).

Bloque 3

Documentación y deuda

Documentación

No es un libro

Es una referencia rápida

La documentación debe responder 3 preguntas en menos de 30 segundos. Si necesita más, es demasiado compleja. Error común: escribir demasiado. Si tu documentación necesita documentación, es demasiado compleja.

3 preguntas en 30 segundos

  • ¿Qué es esto?
  • ¿Cómo lo uso?
  • ¿Qué NO debo hacer?
¿Qué es esto? → Nombre + propósito en una oración. "Button: permite al usuario ejecutar una acción." ¿Cómo lo uso? → Variantes + propiedades + ejemplo visual. "Primary para acción principal, secondary para secundaria." ¿Qué NO debo hacer? → Do/Don't con ejemplos. "Don't: 3 botones primarios juntos." Para este curso: documentación dentro de Figma es suficiente. Página dedicada con componentes organizados.

Concepto

Deuda de diseño

Decisiones pospuestas conscientemente

Así como existe deuda técnica, existe deuda de diseño. La clave: que sea consciente y documentada. Tener deuda es profesional. Tener deuda invisible es peligroso. Tipos: componentes faltantes, estados incompletos, responsive pendiente, accesibilidad parcial.

Tipos de deuda

  • Componentes faltantes
  • Estados incompletos
  • Responsive pendiente
  • Accesibilidad parcial
Componentes faltantes: "No tenemos tabla porque el MVP no tiene tablas. Si se agrega, hay que diseñarla." Estados incompletos: "El upload no tiene estado de error de red. Se agrega en v2." Responsive pendiente: "El dashboard solo está para desktop. Móvil es deuda." Accesibilidad parcial: "Contrastes verificados, pero orden de tabulación no." Formato: qué falta + por qué se pospuso + cuándo se debería resolver. Preguntar: "¿Qué decidiste NO hacer y por qué?"
Bloque 4

Criterios de aceptación

Repaso

Dado / Cuando / Entonces

Aplicado a componentes

En clase 01 vimos criterios de aceptación para flujos. Hoy los aplicamos a componentes individuales. El criterio es el contrato entre diseño y desarrollo. Elimina ambigüedad. Si no está en los criterios, no se puede reclamar que falta.

Ejemplo · Button

  • Dado estado default → hover → fondo cambia con 150ms ease
  • Dado estado loading → clic → no se dispara acción
  • Dado tipo destructive → clic → modal de confirmación
Primer criterio: define la transición visual exacta. No "cambia de color" sino "fondo cambia a color/primary/hover con transición de 150ms ease". Segundo criterio: previene doble envío. Mientras está en loading, el botón no responde a clics. Tercer criterio: acciones destructivas siempre piden confirmación. No se ejecutan directamente. Estos criterios son específicos y testeables. "El botón se ve bien" NO es un criterio.

Ejemplo · Input

  • Dado error de validación → corrige y blur → error desaparece
  • Dado tipo password → clic en ojo → texto se muestra/oculta
Primer criterio: el error desaparece cuando el usuario corrige el valor y sale del campo. No se queda pegado. Segundo criterio: el toggle de visibilidad de contraseña alterna entre mostrar y ocultar el texto. Estos criterios cubren interacciones que un desarrollador podría implementar de muchas formas. El criterio define la forma correcta.
